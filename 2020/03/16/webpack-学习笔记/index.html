webpack的执行过程

  • 正常打包过程
    1. 执行npx webpack
    2. 检查是否有webpack.config.js，如果有就默认执行webpack.config.js里的配置，如果没有，则默认在dist里创建main.js
    3. 寻找src里面的index.js文件
    4. 编译完后，会把编译完的文件放在dist里面
  • 遇到JS引用了非JS文件
    1. 发现不是JS文件，去配置文件里面查找有无第三发的loader
    2. 如果找到，就会去调用相对应的loader来处理这个文件
    3. 当调用loader时，是从后往前调用的
    4. 当最后一个loader调用完毕，会把处理的结果，交给webpack来打包合并

question and answer

  1. 每次都要在cmd里执行webpack sourceFileDir targetFileDir,很麻烦有没有办法解决？

    • 可以，在项目根目录下创建webpack.config.js文件，在文件内配置基本的数据

  2. webpack-dev-server

    1. 能不能在sourceFile一保存时就自动执行webpack指令

      • 可以，需要在项目安装webpack-dev-server插件

    2. 为什么webpack-dev-server执行是放在内存里的

      • 因为速度快，每次放到磁盘会损害磁盘

    3. 那什么时候会放到磁盘里

  3. html-webpack-plugin

    1. 那既然会损耗磁盘，那html可不可以也放在内存
      • 通过html-webpack-plugin也可以

  4. style-loader and css-loader

    1. 既然JS文件里可以引入其他JS，那能否引入CSS文件来减少请求
      • 通过style-loader和css-loader可以实现

webpack-dev-server

  • warn：

    1. 运行npm install webpack-dev-server -D把这个工具安装到本地项目依赖
    2. 安装完毕后，这个工具的用法，和webpack的命令一样
    3. 由于，我们是项目中安装本地的，所以不能使用cmd执行webpack-dev-server命令
    4. 需使用npm run 命令来执行（自行配置到配置文件里添加命令）
    5. webpack-dev-server帮我们打包的js，并不是保存到磁盘里的，而是保存到内存里的
    6. webpack-dev-server打包的文件，会以一种虚拟的方式放在根目录下

  • webpack-dev-server的参数

    • – open
      • 执行即打开borwser
    • –port 8080
      • 修改端口
    • –contentBase dir
      • 修改默认打开路径
    • –hot
      • 异步刷新页面

常用的loader&module

html-webpack-plugin

  • 使用：
    1. 在webpack配置文件中，导入html-webpack-plugin模块
    2. 在plugins数组中new 这个实例
    3. 指定实例的两个属性
      • template // 模板文件
      • filename // 生成页面的名称

CSS类型的包

  • style-loader and css-loader
    • webpack用来处理CSS文件,在module中配置
    • 在css-loader中有几个常用的参数
      1. modules：CSS的模块化，在React中会用到
      2. localIdentName： 指定Class的名字，防止重名
    • 在模块化之后可以用:global(.className)来暴露全局对象，但是他就不在私有属性里面了
  • less-loader
    • webpack用来处理CSS文件,在module中配置
    • 注意：他需要依赖style-loader和css-loader
    • 内部依赖less框架
  • sass-loader
    • webpack用来处理CSS文件,在module中配置
    • 注意：他需要依赖style-loader和css-loader
    • 内部依赖node-sass框架
      • npm install node-sass下载不了，会被墙掉
      • 使用npm i node-sass --sass_binary_site=https://npm.taobao.org/mirrors/node-sass/

图片类型的包

当在CSS文件里面引入了路劲文件时，webpack解析不了，

  • url-loader

    • options对象来指定
      • limit: num | boolean | string
        • num：当文件类型大于num时，不会被转换成base64编码
        • boolean：是否要转换为base64编码
        • string：什么类型的文件转换成base64编码
      • [name]= [hash:8]-[name].[ext]
        • 将不被转换成base64的文件,进行文件名字处理
        • 导入的图片名称会被转换成hash值，防止名字重复的现象
        • 如果让名称不变的话，会把所有同名的图片都变成最后导入的那张
        • 如果想要名字不变，又不会导入错图片，只能在名字前面加少量的hash值

  • esModule

    • 内部依赖file-loader

import

  • import的执行过程(import Vue from "vue")
    1. 找到项目根目录的node_modules文件夹
    2. 根据包名，找到对应的vue文件夹
    3. 在vue文件夹中，找一个叫做package.json的配置文件
    4. 在package.json文件中，查找一个main属性[main属性指定了包在被加载时的入口文件]
      • vue的package中main指向的是vue.runtime.common，阉割版的vue。如果按vue那样正常使用会报错，有三种方法解决问题
        1. 修改vue的package中的main指定的路径
        2. import Vue from "dist/vue.js",把路径写全
        3. 在webpack.config.js中添加命令

部署阶段

  1. 在开发阶段时，打包的文件会非常大，开发和部署不是一个概念
    • 开发阶段用webpack.config.js;
    • 部署阶段用webpack.production.config.js
    • 在配置文件package中指定script指令，让webpack的config属性指向webpack.production.config.js
  2. 重新加载文件，每次打包前都先删除旧dist
    • clean-webpack-plugin
    • warn:他的导出方法是按需导出的，也就是代码可能是exports var CleanWebpackPlugin,所以引入时也需要按需导入
    • 不指定参数的情况下，他会默认清除output指定的文件
  3. 抽离公共代码
    • 在webpack打包时，会把第三方包和自己的代码一起打包到一个文件，这样固然不好，分开来会非常舒服
    • warn:webpack3和webpack4中的抽离是由一点区别的，webpack4把CommonsChunkPlugin这个API给删除了，取而代之的是optimization
